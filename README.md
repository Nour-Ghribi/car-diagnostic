# car-diagnostic

AI diagnostic layer for an automotive diagnostic system with:
- prompt rewriting
- semantic retrieval
- bounded goal resolution and planning
- mock-backed OBD runtime tools

## Branch to use

This work was pushed on:

```text
codex/csv-business-layer-runtime-mocks
```

## What the project does

The project takes a user automotive question such as:

```text
"check my car health"
"show rpm"
"read dtc"
"why is my battery weak"
```

and produces a grounded diagnostic response with:
- public intent
- internal goal
- execution plan
- evidence
- recommendations

Important:
- the NLP / retrieval / resolver pipeline is real
- the OBD runtime data is still mocked

## Main input

The core runtime input is `AgentRequest` from [backend/schemas.py](./backend/schemas.py).

Minimal useful input:

```json
{
  "request_id": "req_001",
  "vehicle_id": "veh_002",
  "user_prompt": "check my car health"
}
```

Main fields:
- `request_id`: caller correlation id
- `vehicle_id`: mock vehicle scenario id such as `veh_001`, `veh_002`, ...
- `user_prompt`: the user question
- `ts`: generated by caller in code paths that use `AgentRequest` directly
- `locale`: optional, defaults to `fr`

## Main output

The core runtime output is `AgentResponse` from [backend/schemas.py](./backend/schemas.py).

Important output fields:
- `intent.name`
- `intent.parameters.goal`
- `diagnosis`
- `confidence`
- `signals_used`
- `actions_taken`
- `missing_data`
- `recommendations`

## Project architecture

High-level pipeline:

```text
User Prompt
-> LLM Rewriter
-> Semantic Retrieval
-> Goal Resolver / Planner
-> Validation
-> Orchestrator
-> Mock OBD Tools
-> Final AgentResponse
```

Business knowledge now comes mainly from:
- [data/diagnostic_profiles.py](./data/diagnostic_profiles.py)
- [data/goal_catalog.py](./data/goal_catalog.py)
- [data/signal_mapping.py](./data/signal_mapping.py)

Runtime mock layer remains separate:
- [data/mock_runtime_scenarios.py](./data/mock_runtime_scenarios.py)
- [agent/tools.py](./agent/tools.py)
- [fake_data/tool_responses/vehicle_profiles.json](./fake_data/tool_responses/vehicle_profiles.json)

## Dependencies

There is no packaged dependency file in the repo yet, so install the needed libraries manually if they are missing.

Core libraries used by the project:
- `pydantic`
- `pytest`

Optional providers:
- `sentence-transformers` for SBERT embeddings
- `openai` for OpenAI or OpenRouter calls
- `google-genai` for Gemini calls

Example:

```powershell

```

```powershell

```

## Minimal setup for a teammate

If your colleague just wants the project to run locally in a stable way, no API key is required.

Recommended minimal PowerShell setup:

```powershell
$env:INTENT_EMBEDDING_PROVIDER="mock"
$env:INTENT_REWRITER_PROVIDER="mock"
$env:INTENT_RESOLVER_PROVIDER="mock"
```

This mode:
- works offline
- does not require OpenAI / Gemini / OpenRouter keys
- keeps OBD runtime mocked

## About the CSV file

The business-knowledge CSV file was **not committed** to the repo.

Why:
- it is treated as an external data source
- it may be local or proprietary

If your colleague has the CSV locally, they can point the project to it with:

```powershell
$env:DIAGNOSTIC_PROFILE_CSV_PATH="C:\\path\\to\\DiagnosticProfile (1).csv"
```

If they do **not** have the CSV:
- the project still runs
- the runtime mock layer still works
- but CSV-derived business enrichment will be missing or reduced

## Available mock vehicle scenarios

Runtime mock scenarios currently include:
- `veh_001`: combustion / misfire tendency
- `veh_002`: healthy baseline
- `veh_003`: lean fuel-control condition
- `veh_004`: cooling / overheating
- `veh_005`: sparse cache requiring refresh
- `veh_006`: electrical low voltage
- `veh_007`: rich-running condition

## Important limitation

The project is **not yet connected to real OBD hardware**.

So:
- final responses are structurally real
- runtime observations are still mock-backed
